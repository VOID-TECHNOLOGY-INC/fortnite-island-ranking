# Fortnite Island Ranking Dashboard v2 機能仕様書

## 1. この仕様書の目的

本書は、既存の `doc/spec.md`、`README.md`、および現行コードベースを踏まえ、`Fortnite Island Ranking Dashboard` を「もっとユーザーにとって便利で使いやすいアプリケーション」に改善するための v2 仕様を定義する。

現状のアプリは以下に強みがある。

- Fortnite Island の一覧を素早く見られる
- 島コード検索ができる
- 各島の AI リサーチ結果を見られる
- 日本語/英語を切り替えられる

一方で、ユーザーが本当に知りたい以下の問いに対する答えはまだ弱い。

- 今どの島が伸びているのか
- その島はなぜ伸びているのか
- 自分が見ている島は前回より良くなったのか
- 他の島と比べて強い点・弱い点は何か

v2 では、単なる一覧ビューアから、意思決定に使える「発見・比較・追跡ダッシュボード」へ進化させる。

## 2. 現状整理

### 2.1 現在実装されている主機能

- ホーム画面
  - ランキング一覧
  - 検索
  - 期間切替（10m / 1h / 24h）
  - 手動リフレッシュ
- 詳細画面
  - Perplexity を使った AI リサーチ本文
  - 出典リンク
- API
  - `/api/islands`
  - `/api/islands/:code/metrics`
  - `/api/islands/:code/research`
  - `/api/top-islands`
- バックエンド
  - Firebase Functions + Express
  - メモリキャッシュ
  - Scheduler によるトップ島キャッシュ温め

### 2.2 現状の主要課題

1. ランキングが「今の一覧」に寄っており、「変化」が見えない
2. 詳細画面が AI リサーチ中心で、定量データの確認導線がない
3. 比較機能が未実装で、意思決定に必要な相対評価ができない
4. フィルタ・ソート・検索の UI が不足し、探索効率が低い
5. 「HypeScore」が UI 文言上は存在するが、実体が不透明で信頼しづらい
6. モバイルでは最低限見られるが、一覧中心で情報の優先度が高くない
7. 継続利用を支える watchlist、最近見た島、共有可能 URL などがない
8. API 制約や検索仕様が UI から分かりにくく、空結果の理由が伝わらない

## 3. 第一性原理で見た改善方針

### 3.1 問題の本質

ユーザーの本質的な仕事は「島の人気そのものを見ること」ではなく、「次に注目すべき島と、その理由を短時間で判断すること」である。

### 3.2 成功条件

以下を 1 画面または 2 クリック以内で判断できること。

- いま強い島
- 伸びている島
- 気になる島の定量推移
- 他島との比較結果
- AI リサーチと定量データの整合

### 3.3 見直すべき前提

| 前提 | 見直し | 方針 |
| --- | --- | --- |
| ランキング一覧があれば十分 | 一覧だけでは判断できない | 変化量と理由を同時に見せる |
| 詳細ページは AI 要約が中心でよい | 要約だけでは信用しづらい | KPI と時系列を主役にし、AI は補助にする |
| ソートはあとから追加すればよい | 探索体験の質に直結する | v2 の中心機能として整理する |
| 比較は将来対応でよい | 比較できないと意思決定に使えない | v2 の必須機能にする |
| 7日分 API があれば十分 | 継続観測には足りない | 長期保存は将来拡張を前提に設計する |

### 3.4 変えない前提

- データの一次ソースは Fortnite Data API を継続利用する
- 既存の Firebase Hosting / Functions 構成は維持する
- 日本語/英語対応は継続する
- AI リサーチは残すが、補助情報として位置付けを見直す

## 4. v2 プロダクト方針

v2 は以下の 4 原則で設計する。

1. 一覧より先に「判断材料」を見せる
2. 定量データを主役、AI を補助にする
3. 比較と追跡を日常動作として成立させる
4. URL 共有とモバイル閲覧を前提にする

## 5. 改善点一覧

### 5.1 情報設計

- ホームを「Top Islands の表」から「ダッシュボード」に再構成する
- 画面上部で `Top` `Rising` `Watchlist` の 3 つの問いに答える
- 詳細画面で AI リサーチより先に KPI サマリーと推移を表示する

### 5.2 探索性

- ソート UI を実装する
- タグ、クリエイター、指標別フィルタを追加する
- 検索候補、最近の検索、クエリ URL 同期を追加する
- 2文字未満の検索条件では理由付きの補助文言を表示する

### 5.3 判断支援

- HypeScore を実装し、算出方法を明示する
- 前回比、24h 比、急上昇バッジを追加する
- 比較機能を実装し、最大 4 島の相対比較を可能にする
- 関連島表示を追加し、次の探索先を提示する

### 5.4 継続利用

- Watchlist を追加し、気になる島を保存できるようにする
- 最近見た島を localStorage に保存する
- 共有 URL と compare URL を生成できるようにする
- 将来の通知機能を見据えてデータ構造を整理する

### 5.5 使いやすさ

- モバイルでカード UI を優先表示する
- コピー成功トースト、読み込みスケルトン、空状態ガイドを追加する
- 更新時刻、データ範囲、API 制約を明確に表示する
- i18n 文言を整理し、不自然な表記や未使用文言を解消する

## 6. スコープ

### 6.1 v2 の対象

- ダッシュボード再設計
- 詳細画面の KPI / チャート強化
- 比較機能

### 6.2 v2 の対象外

### 7.1 想定ユーザー

- Fortnite クリエイター
- 競合リサーチ担当
- UEFN の分析用途ユーザー
- SNS / コミュニティ運営者

2. 気になる島の推移を見たい
3. 自分の島と人気島を比較したい
4. 同じ系統の島を横断して見たい
5. あとでまた見る島を保存したい

## 8. 画面仕様

### 8.1 ホーム `/`

#### 目的

最初の 10 秒で「何を見るべきか」を理解できるようにする。

#### 画面構成

1. ヘッダー
   - ロゴ
   - 言語切替
   - Watchlist 入口

2. グローバルフィルタバー
   - 期間切替: `10m` `1h` `24h`
   - 検索ボックス
   - ソート切替
   - タグフィルタ
   - クリエイターフィルタ
   - `Reset` ボタン

3. サマリーエリア
   - `Top Islands`
   - `Rising Now`
   - `Most Retained`
   - `Most Recommended`

4. メインランキング
   - 表示切替: `Table` / `Cards`
   - 列: Rank, Island, Creator, HypeScore, Unique Players, Peak CCU, Delta, Tags
   - 行操作: `詳細を見る`, `比較に追加`, `Watchlist に追加`, `コードをコピー`

5. 補助エリア
   - 最近見た島
   - 検索候補
   - API 更新時刻
   - データ範囲説明

#### 動作要件

- すべてのフィルタ状態を URL クエリに保存する
- URL を再訪した際に同じ状態を復元する
- 検索入力は 300ms デバウンスで反映する
- クエリが 2 文字未満の場合は「検索対象を絞るには 2 文字以上必要」と表示し、通常ランキングを維持する
- ランキングは `Top` と `Rising` をタブで切り替えられる

#### ソート仕様

- `HypeScore`
- `Unique Players`
- `Peak CCU`
- `Minutes / Player`
- `D1 Retention`
- `Recommends`
- `Favorites`
- `Latest Change`

#### 空状態

- データなし時は原因別メッセージを表示する
  - 検索条件が厳しい
  - API データなし
  - 一時的な取得失敗

### 8.2 詳細 `/island/:code`

#### 目的

1 島の現状、推移、評価理由を 1 画面で把握できるようにする。

#### 画面構成

1. ヒーローエリア
   - 島名
   - 島コード
   - クリエイター名
   - タグ
   - `比較に追加`
   - `Watchlist に追加`
   - `共有`

2. KPI サマリーカード
   - HypeScore
   - Unique Players
   - Peak CCU
   - Minutes / Player
   - D1 Retention
   - D7 Retention
   - Recommends
   - Favorites

3. 変化サマリー
   - 直近差分
   - 24h 差分
   - 急上昇 / 急落バッジ
   - 前回更新時刻

4. チャートエリア
   - 指標切替式の時系列チャート
   - `10m` `1h` `24h` の再切替
   - 指標の重ね表示は最大 2 系列まで

5. AI リサーチエリア
   - 要約
   - 出典
   - `再生成`（明示操作時のみ。レート制御付き）
   - `最終取得時刻`

6. 関連島エリア
   - 同タグ
   - 同クリエイター
   - 同系統の上位島

#### 動作要件

- ページ初回表示時に KPI サマリーと最新チャートを優先表示する
- AI リサーチは遅延読込し、KPI より後に描画してよい
- `共有` を押すと現在の `window` を含む URL をコピーする

### 8.3 比較 `/compare`

#### 目的

複数島の強み・弱みを短時間で比較できるようにする。

#### 画面構成

- 比較対象一覧
- 指標比較表
- HypeScore 分解表示
- 折れ線チャート
- レーダーチャート
- 差分コメント

#### 制約

- 比較対象は最大 4 島
- 2 島未満では比較 UI を表示せず、追加導線付きの空状態を表示する
- 比較対象は URL クエリを正とし、`codes` 未指定時のみ localStorage から直前の選択を復元する

#### 指標比較項目

- HypeScore
- Unique Players
- Peak CCU
- Minutes / Player
- D1 / D7 Retention
- Recommends
- Favorites

### 8.4 Watchlist

#### v2 での実装方針

- ログインなし
- localStorage 保存
- 最大 30 島
- 並び順は手動変更不可、追加順の逆順

#### 表示内容

- 島名
- 直近 HypeScore
- 24h 変化量
- 前回閲覧日時

#### 主な動作

- ホーム、詳細、比較から追加できる
- 一覧から削除できる
- 共有 URL に watchlist 自体は含めない

## 9. 指標・スコア仕様

### 9.1 HypeScore の位置付け

HypeScore は「人気の大きさ」ではなく、「今注目すべき度合い」を表す補助指標とする。

### 9.2 入力指標

- Unique Players
- Peak CCU
- Minutes / Player
- D1 Retention
- Recommends

### 9.3 基本ルール

- 指標ごとに同一 window 内で正規化する
- 欠損値は 0 ではなく「未評価」として扱い、重みを再配分する
- UI 上で「β」表記を付け、算出ロジックへのリンクを表示する

### 9.4 表示ルール

- スコア本体
- 前回比
- 24h 比
- 内訳ツールチップ

## 10. API 仕様

### 10.1 既存 API の扱い

- `/api/islands`
  - 検索専用 + 軽量一覧取得として継続
- `/api/islands/:code/metrics`
  - 詳細/比較用として継続
- `/api/islands/:code/research`
  - AI 補助情報として継続
- `/api/top-islands`
  - 後方互換の「人気島一覧」API として継続する
  - v2 の `dashboard` / `overview` / `compare` は、このレスポンスを直接つなぐのではなく、同じ集約ロジックを共有する

### 10.2 追加 API

#### `GET /api/dashboard`

ホーム画面用の集約 API。

クエリ:

- `window=10m|1h|24h`
- `query=<text>`（任意）
- `sort=hype|unique|peak|mpp|d1|recommends|favorites|change`（任意、`ranking` にのみ適用）
- `tags=tag-a,tag-b`（任意）
- `creator=<name>`（任意）

適用ルール:

- `window` は `ranking` / `rising` / `highRetention` / `highRecommend` の全セクションに適用する
- `query` / `tags` / `creator` は候補集合を同じ条件で絞り込み、全セクションに適用する
- `sort` は `ranking` の並び順だけを変更し、`rising` / `highRetention` / `highRecommend` の意味は固定する
- `tab` / `view` は URL には保存してよいが、UI 状態なので API クエリには含めない

レスポンス例:

```json
{
  "window": "10m",
  "updatedAt": "2026-03-06T12:00:00.000Z",
  "ranking": [],
  "rising": [],
  "highRetention": [],
  "highRecommend": [],
  "degraded": false
}
```

配列要素の共通形:

- `code`, `name`, `creator`, `tags`
- `hypeScore`
- `kpis`（`uniquePlayers`, `peakCcu`, `minutesPerPlayer`, `retentionD1`, `retentionD7`, `recommends`, `favorites`）
- `deltas`（直近差分、24h 差分）

#### `GET /api/islands/:code/overview`

詳細画面の初回描画に必要な情報をまとめて返す。

クエリ:

- `window=10m|1h|24h`（省略時は `10m`）

レスポンス項目:

- basic
- kpis（指定 `window` 時点の最新値）
- deltas（指定 `window` に対する直近差分 + 24h 差分）
- tags
- related（指定 `window` の文脈で算出）
- researchStatus

補足:

- `basic` と `researchStatus` は window 非依存でよい
- 共有 URL で復元する `window` と overview の `window` は同一値を使う

#### `GET /api/compare`

クエリ:

- `codes=1111-2222-3333,4444-5555-6666`
- `window=10m`
- `codes` は 2〜4 件を受け付ける

レスポンス:

- island summaries
- metric series
- normalized scores

### 10.3 API 設計方針

- ホーム初回描画は 1 リクエストを基本とする
- `dashboard` / `overview` / `compare` は共通の island summary / normalization layer を使って組み立てる
- キャッシュキーは `window` `query` `sort` `tags` `creator` を含める
- `tab` / `view` は URL で保持してよいが、クライアント UI 状態なので API キャッシュキーには含めない
- レート制限時は部分成功を許容し、取得できたデータを返す
- エラー時も UI が崩れない構造を維持する

## 11. フロントエンド要件

### 11.1 状態管理

- SWR を継続利用する
- URL クエリを単一の画面状態ソースとして扱う
- localStorage は watchlist、最近見た島、言語設定のみに限定する

### 11.2 コンポーネント方針

- Home 用カード群を独立コンポーネント化する
- RankingTable を `table` と `cards` の 2 表示に分割する
- Detail 用 KPI カードとメトリクスチャートを共通化し、Compare でも再利用する

### 11.3 モバイル要件

- 640px 未満ではランキングをカード表示優先にする
- フィルタはドロワー化する
- KPI カードは 2 列グリッドにする

## 12. バックエンド要件

### 12.1 キャッシュ

- 現行メモリキャッシュは継続
- `10m` は 5〜10 分
- `1h` は 10〜15 分
- `24h` は 15 分
- dashboard 系は warm 対象とする

### 12.2 データ整形

- 生の Fortnite API レスポンスはフロントへ極力そのまま返さない
- 欠損値、空系列、部分失敗を正規化して返す
- HypeScore 算出はサーバー側で行う

### 12.3 将来拡張

- 日次または 10 分単位のスナップショット保存を行える構造にする
- Firestore または BigQuery への退避を v2.1 以降の候補とする

## 13. UX ライティング要件

- データなし時は「何が起きたか」を具体的に伝える
- AI リサーチには「参考情報」であることを明記する
- HypeScore には「参考スコア」であることを明記する
- 期間、更新時刻、データ取得元を常時視認可能にする

## 14. アクセシビリティ要件

- 主要操作はキーボード操作可能
- コントラスト比は WCAG AA を満たす
- テーブルには見出し、ボタンには明確な `aria-label` を付与する
- コピー結果、エラー、更新成功はスクリーンリーダーへ通知する

## 15. 計測要件

以下のイベントを計測対象とする。

- 検索実行
- フィルタ変更
- 詳細遷移
- Compare 追加
- Watchlist 追加/削除
- 共有 URL コピー
- AI リサーチ再生成

主要指標:

- ホーム離脱率
- 詳細遷移率
- Compare 利用率
- Watchlist 保存率
- 再訪率

## 16. 受け入れ条件

### 16.1 ホーム

- ユーザーは 10 秒以内に `Top` と `Rising` を見分けられる
- ソート、検索、フィルタの状態が URL に反映される
- 空状態で原因別メッセージが表示される

### 16.2 詳細

- KPI サマリーが AI リサーチより先に表示される
- 1 指標以上のチャートを必ず表示できる
- 共有 URL から同じ `window` 状態で再表示される

### 16.3 比較

- 2〜4 島を追加して比較できる
- 比較対象はページ再読込後も維持される

### 16.4 Watchlist

- ホームと詳細の両方から追加できる
- 保存済み島が再訪時に残る

## 18. 補足

本仕様は、現行コードベースの延長で無理なく実装できることを重視している。特に以下を前提とする。

- 既存の React + SWR + React Router 構成を維持する
- 既存の Firebase Functions API を拡張して段階的に移行する
- AI リサーチは削除せず、定量データ中心の画面に再配置する

これにより、現行アプリの資産を活かしつつ、ユーザー体験の中心を「一覧を見る」から「判断して次の行動を決める」へ移せる。
